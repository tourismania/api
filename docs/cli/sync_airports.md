# CLI-команда `airports sync`

> **Файлы:** [handler.go](../../internal/application/command/sync_airports/handler.go), [wikidata/client.go](../../internal/infrastructure/geo/wikidata/client.go), [mwgg/client.go](../../internal/infrastructure/geo/mwgg/client.go), [static/countries_ru.go](../../internal/infrastructure/geo/static/countries_ru.go), [cli/airports.go](../../internal/presentation/cli/airports.go)

Команда наполняет таблицы `countries`, `cities`, `airports` данными из внешних источников. Запускается вручную раз в месяц админом (см. `README.md → CLI`).

```bash
go run ./cmd/cli airports sync --dry-run   # только посчитать, не писать в БД
go run ./cmd/cli airports sync             # полная синхронизация
```

---

## 1. Зачем нужны какие данные

| Данные | Источник | Зачем |
|---|---|---|
| ICAO, IATA, название аэропорта, координаты, высота, часовой пояс, город/регион/страна (на английском) | [mwgg/Airports](https://github.com/mwgg/Airports) (GitHub JSON, ~40 000 записей) | Единственный бесплатный публичный дамп с полным покрытием аэропортов мира. Это основа: без него не из чего строить `airports`/`cities`/`countries`. |
| Русское название аэропорта | Wikidata SPARQL | Проект — русскоязычный сервис для путешественников; показывать "Sheremetyevo International Airport" вместо "Шереметьево" неприемлемо для UX. |
| Русское название города | Wikidata SPARQL | Аналогично — город показывается пользователю, должен быть на русском. |
| Русское название страны | Статическая карта `static/countries_ru.go` (~250 стран) | Стран — конечный, стабильный набор (не меняется месяцами). Гонять SPARQL-запрос ради 250 записей, которые не меняются, — не нужно; проще и надёжнее держать статическую таблицу. |

mwgg отдаёт `city`/`state`/`country` только на английском — это ожидаемо, так как источник не специализирован под русскоязычные проекты. Поэтому дообогащение через Wikidata обязательно для двух из трёх сущностей (аэропорт, город), но не для страны.

---

## 2. Алгоритм по шагам

```
1. Fetch аэропортов                     → mwgg.Client.Fetch(ctx)
2. Сортировка по ICAO                   → детерминированный порядок для всех последующих шагов
3. Fetch RU-названий аэропортов         → wikidata.Client.FetchAirportNamesRU(ctx)   (по ICAO, P239)
4. Собрать уникальные (город, страна)   → collectCityRefs(records)
5. Fetch RU-названий городов            → wikidata.Client.FetchCityNamesRU(ctx, cityRefs)
6. Upsert стран                         → CountryRepository.Upsert (RU-имя из static-карты)
7. Upsert городов (дедуп по city+state+country) → CityRepository.Upsert (RU-имя, если есть перевод, иначе EN)
8. Upsert аэропортов                    → AirportRepository.Upsert (RU-имя, если есть перевод, иначе EN; city_id из шага 7)
```

В `--dry-run` шаги 6–8 не пишут в БД, но фактически выполняют весь путь получения данных (включая оба Wikidata-запроса), чтобы посчитать честную статистику `countries`/`cities`/`airports`.

### Почему город переводится не по ICAO, а по (имя, страна)

Раньше `FetchCityNamesRU` принимала ICAO аэропорта и определяла русское название города через связь `wdt:P131` ("расположен в административно-территориальной единице") **у конкретного аэропорта**. Это ломалось на практике:

- Многие аэропорты физически находятся за городской чертой (в области/районе), поэтому P131 у них указывает на что угодно — район, область, посёлок, — а не на сам город. Пример: у аэропорта Емельяново (`UNKL`, Красноярск) P131 в Wikidata ведёт прямиком в «Красноярский край», а не в «Красноярск».
- Handler дополнительно группировал все аэропорты с одинаковым `(city, state, country)` (например, все аэропорты Москвы и области — `UUBB, UUBM, UUDD, UUEE, UUMO, UUMU`) и брал перевод **только у первого по алфавиту ICAO в группе** — это `UUBB` (аэропорт Быково). У Быково P131 корректно вело в посёлок Быково, но этот перевод "протекал" на Домодедово, Шереметьево и остальные — в БД сохранялся город «Быково» вместо «Москва».

Сейчас перевод ищется напрямую по `CityRef{Name, CountryISO2}` — то есть по _названию города из источника_, а не по тому, "где Wikidata считает физически расположенным этот конкретный аэропорт". Это структурно исключает обе проблемы: перевод города больше никогда не зависит от того, какой аэропорт его запросил.

## 3. Как устроен запрос к Wikidata за русским названием города

Файл: `wikidata/client.go`, функция `FetchCityNamesRU`.

### 3.1 Шаг 1 — таблица ISO2 → Wikidata QID страны

```sparql
SELECT ?country ?iso2 WHERE { ?country wdt:P297 ?iso2 . }
```

Один запрос на весь прогон, ~260 строк. Нужен, чтобы в основном запросе привязывать страну **напрямую по QID** (`wd:Q159` для России), а не через обратный поиск по ISO2-коду.

**Почему не искать страну по ISO2 прямо в основном запросе.** Изначально основной запрос содержал `?country wdt:P297 ?iso2 .` внутри той же секции, что и поиск города по имени. Это оказалось катастрофически медленным на публичном эндпоинте WDQS: запрос с батчем из ~150 названий городов уходил в таймаут (HTTP 504), независимо от того, был ли ISO2 связан литералом или через join с переменной. Причина — WDQS плохо строит план выполнения, когда обратный поиск по редко используемому свойству (P297) комбинируется с большим `VALUES` по строковым меткам. Прямая привязка по QID (`wd:Q159`) убирает эту эту проблему полностью: тот же батч из 150 названий стабильно отрабатывает за ~0.5–1.5 секунды.

### 3.2 Шаг 2 — batched-запрос городов на одну страну

```sparql
SELECT DISTINCT ?nameEn ?cityRu WHERE {
  VALUES ?nameEn { "Moscow"@en "Krasnoyarsk"@en ... }
  ?city rdfs:label ?nameEn .
  ?city wdt:P17 wd:Q159 .
  ?city wdt:P625 ?coord .
  ?city rdfs:label ?cityRu .
  FILTER(LANG(?cityRu) = "ru")
}
```

Города группируются по `CountryISO2` (`groupByCountry`), внутри страны бьются на батчи по `cityBatchSize = 100` названий (`chunkStrings`). Страны, для которых в шаге 3.1 не нашёлся QID, пропускаются целиком — их города остаются с английским названием (штатный fallback, ошибкой не считается).

**Почему совпадение по точной английской метке, а не по типу "населённый пункт".** Проверялся вариант с фильтром `?city wdt:P31/wdt:P279* wd:Q486972` ("является — транзитивно — населённым пунктом"), чтобы отсекать не-города. Он корректен по смыслу, но дорог: транзитивное замыкание `P279*` на батче в 150 названий отрабатывало 24+ секунды (иногда — таймаут). При этом точное совпадение по метке само по себе уже отсекает большинство ложных срабатываний: «Krasnoyarsk Krai» — это другая строка, чем «Krasnoyarsk», поэтому регион никогда не попадёт в выдачу для поиска "Krasnoyarsk".

**Почему всё-таки нужен `wdt:P625` (наличие координат).** Точное совпадение по метке не спасает от омонимов: у Wikidata нашлась подводная лодка «К-173 «Красноярск»», у которой английская метка тоже ровно `"Krasnoyarsk"@en`. Без доп. фильтра ответ на "Krasnoyarsk" мог случайно вернуть перевод от подлодки вместо города (порядок строк в ответе SPARQL не гарантирован, а код просто перезаписывает значение в map — `out[key] = value`). Требование `?city wdt:P625 ?coord .` (наличие географических координат) — дешёвый однохоповый фильтр, который эту подлодку отсекает: у неё нет `P625`, у любого настоящего города — есть. Проверено, что добавление этого условия не замедляет запрос (тот же батч из 150 названий — по-прежнему ~1.5 секунды).

### 3.3 Экранирование

Названия городов подставляются в `VALUES` как строковые литералы SPARQL, поэтому перед вставкой экранируются кавычки и обратные слэши (`escapeSPARQLString`) — иначе название вроде `Cote d"Ivoire`-подобного (гипотетически, с кавычкой) сломало бы синтаксис запроса.

## 4. Тайминги (замерено вручную на живом `query.wikidata.org/sparql`)

| Запрос | Время |
|---|---|
| Таблица ISO2 → QID страны (шаг 3.1, ~260 строк) | ~0.6 с, один раз за прогон |
| Батч из 100–150 названий городов на одну страну, с `P625`-фильтром (шаг 3.2) | ~0.5–1.5 с |
| То же самое, но с обратным поиском `?country wdt:P297 ?iso2` вместо QID | таймаут (504) уже на ~150 названиях — этот вариант не используется |
| То же самое, но с `wdt:P31/wdt:P279*` вместо `P625` | 24+ с на батч из 150 названий — не используется |
| RU-названия аэропортов (`FetchAirportNamesRU`, постраничный запрос по 10 000 записей) | не менялось этой правкой; несколько запросов на весь набор аэропортов |

Итоговое ожидаемое время полного прогона команды — то же **~3–4 минуты** на ~40 000 аэропортов, что и раньше (см. `README.md`). Количество уникальных городов в источнике — несколько тысяч, поэтому дополнительные батч-запросы по городам не меняют порядок величины общего времени: это по-прежнему разовая ежемесячная админ-задача, где корректность важнее секунд.

## 5. Дедупликация и upsert

- **Города** дедуплицируются по ключу `(lower(name), lower(state), upper(country))` в памяти — несколько аэропортов одного города (например, все аэропорты Москвы) записываются в БД как одна строка `cities`, upsert идёт один раз, остальные аэропорты переиспользуют её `id`. В БД это дополнительно защищено уникальным индексом `cities_uniq_name_state_country` (миграция 011).
- **Страны** — upsert по PK `iso2`.
- **Аэропорты** — upsert по PK `icao`.
- Порядок upsert-ов: страны → города → аэропорты (аэропорт ссылается на `city_id`, город — на `country_iso2`).

---

## 6. Известные ограничения

- **Омонимичные города внутри одной страны.** Если в стране есть несколько разных населённых пунктов с одинаковым английским названием (в России, например, есть несколько сёл «Быково» в разных областях), запрос вернёт перевод от какого-то одного из них — какого именно, не гарантировано. Для целевых кейсов проекта (крупные города — центры аэропортов) это не проявляется; для мелких региональных аэропортов теоретически возможно.
- **Страна без Wikidata QID.** Если для ISO2-кода страны нет записи `wdt:P297` в Wikidata (редкие/спорные территории), все города этой страны остаются без RU-перевода — это осознанный fallback на английское название, а не ошибка выполнения.
- **Retry/backoff.** Все SPARQL-запросы (включая новые из п. 3) используют общий механизм ретраев с экспоненциальной задержкой (3с → 6с → 12с, всего 4 попытки) на 5xx/429/сетевые ошибки — отдельно реализовывать это для городов не потребовалось.
