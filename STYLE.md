## 1. Стиль кода

### Язык и инструментарий

**Основной язык:** Go 1.24  
**Модуль:** `api`

| Параметр | Правило |
|---|---|
| Отступы | Табы (стандарт `gofmt`) |
| Длина строки | До 120 символов; превышение допустимо только если разбивка ухудшает читаемость |
| Фигурные скобки | Открывающая — на той же строке (`Go style`) |
| Точки с запятой | Не используются (автовставка компилятором) |
| Trailing comma | Обязательна в многострочных литералах структур и слайсов |

### Соглашения об именовании

| Элемент | Конвенция | Пример |
|---|---|---|
| Переменные, поля | `camelCase` | `userCreator`, `eventBus` |
| Экспортируемые типы и функции | `PascalCase` | `UserCreator`, `WriteJSON` |
| Неэкспортируемые функции | `camelCase` | `hashPassword` |
| Константы | `PascalCase` (или `iota`-группы) | `MaxRetryCount`, `RoleSuperAdmin` |
| Sentinel-ошибки | `ErrPascalCase` | `ErrUserNotPersisted` |
| Файлы | `snake_case` | `user_creator.go`, `get_me_http_test.go` |
| Тестовые файлы | `<имя_файла>_test.go` | `user_creator_test.go` |
| Пакеты | Короткий lowercase, без подчёркиваний | `entity`, `httpx`, `createuser` |
| Интерфейсы | `PascalCase`, суффикс `-er` если описывает действие | `UserRepository`, `PasswordHasher`, `UseCase` |
| Конструкторы | `New<TypeName>` | `NewUserCreator`, `NewHandler` |

### Форматирование и линтер

- **Форматтер:** `gofmt` / `goimports` — запускается автоматически, ручная правка запрещена
- **Линтер:** `golangci-lint`
- **Конфиг:** корневой `.golangci.yml` (при появлении)
- **Перед коммитом:** `golangci-lint run` должен быть чистым

```bash
# Запуск линтера
golangci-lint run

# Форматирование
gofmt -w ./...
```

## 2. Архитектурные соглашения

### Направление зависимостей

```
Presentation → Application → Domain ← Infrastructure
```

Нарушение этого направления — блокирующий дефект на ревью.

### Правила слоёв

| Слой | Где | Что можно импортировать | Что запрещено |
|---|---|---|---|
| **Domain** | `internal/domain/` | Только stdlib + `github.com/google/uuid` | `pgx`, `chi`, `kafka`, любые инфра-пакеты |
| **Application** | `internal/application/` | `domain/`, stdlib | `pgx`, `chi`, конкретные реализации инфраструктуры |
| **Infrastructure** | `internal/infrastructure/` | `domain/`, stdlib, любые внешние либы | Пакеты `presentation/` |
| **Presentation** | `internal/presentation/` | `application/`, `infrastructure/auth`, stdlib | Прямые вызовы domain-сервисов в обход application |

### CQRS

- **Команды** (write-side): `internal/application/command/<action>/`  
  Каждая команда — отдельный каталог с файлами `command.go`, `handler.go`, `result.go`.
- **Запросы** (read-side): `internal/application/query/<action>/`  
  Аналогичная структура.
- Handler команды не должен содержать бизнес-логики — только делегирование доменному сервису.

### Доменные сущности vs. ORM-модели

`domain/entity.User` и `infrastructure/persistence/postgres/model.User` — **разные типы**. Маппинг между ними живёт в слое инфраструктуры. Никогда не добавляй теги `db:` или `json:` в доменные сущности.

### Интерфейсы репозиториев

Определяются в `domain/repository/`, реализуются в `infrastructure/persistence/`. В тестах использовать ручные моки или `gomock` — **не** конкретные реализации.

## 3. HTTP и API

### URL-соглашения

| Тип | Префикс | Пример |
|---|---|---|
| Публичные эндпоинты | `/api/` | `POST /api/login` |
| Версионированные бизнес-эндпоинты | `/api/v1/` | `GET /api/v1/users/me` |
| Документация | `/api/docs/` | `GET /api/docs/*` |
| Healthcheck | `/healthz` | `GET /healthz` |

- Все бизнес-эндпоинты — только под `/api/v1/` с JWT-мидлваром.
- Регистрация маршрутов — исключительно в `presentation/http/router.go`.
- Новый роут → добавить в раздел **Endpoints** файла `README.md`.

### Формат ответа

Ошибки всегда возвращаются через `httpx.WriteError` / `httpx.WriteValidationError`:

```json
{
  "error": "описание ошибки",
  "code": 400,
  "meta": { "Email": "required" }
}
```

Успешные ответы — через `httpx.WriteJSON`. Структура payload определяется хендлером.

### Валидация входных данных

- Валидация на границе presentation-слоя (`go-playground/validator`).
- В домен попадают уже чистые данные.
- Теги `validate:"..."` добавляются к DTO-структурам в `presentation/`, **не** к доменным сущностям.

## 4. Обработка ошибок

```go
// Правильно: оборачивать с контекстом через %w
return fmt.Errorf("store user: %w", err)

// Правильно: sentinel для различимых ошибок
var ErrUserNotPersisted = errors.New("user was not persisted")

// Неправильно: глотать ошибку молча
_ = repo.Store(ctx, user)

// Неправильно: log.Fatal вне main()
log.Fatal(err)
```

**Правила:**

- Оборачивать ошибки через `fmt.Errorf("контекст: %w", err)` — цепочка сохраняется для `errors.Is/As`.
- Логировать ошибки только на границе (middleware), не на каждом уровне стека.
- `log.Fatal` и `os.Exit` — только в `main()`.
- Типизированные sentinel-ошибки (`var ErrXxx = errors.New(...)`) — для ошибок, на которые нужно реагировать по-разному.
- Никогда не поглощать ошибки молча (`_ = err` допустимо только при явной причине в комментарии рядом).

## 5. Логирование

- **Формат:** структурированный JSON (`log/slog` стандартной библиотеки).
- **Request ID** включается в каждую запись — пробрасывается через `middleware.RequestID`.

| Уровень | Когда использовать |
|---|---|
| `debug` | Внутренние состояния, только во время разработки |
| `info` | Штатные операции (старт сервера, выполненная миграция) |
| `error` | Сбои, требующие внимания |

```go
slog.Error("encode response", "err", err)
slog.Info("server started", "addr", cfg.Addr)
```

## 6. Документация кода

- Все публичные типы, функции и методы — с `godoc`-комментарием.
- Первая строка комментария: краткое предложение, начинается с имени символа.
- Комментарий объясняет **почему**, а не **что** — код сам показывает «что».
- Закомментированный код не коммитится — для истории есть git.

```go
// UserCreator orchestrates registration: hash credentials, persist, then
// publish a UserRegistered event so async consumers can react.
type UserCreator struct { ... }

// Create hashes the user's password, stores the entity, and publishes a
// UserRegistered event. Event-publish failures are returned to the caller
// rather than swallowed.
func (s *UserCreator) Create(ctx context.Context, user entity.User) (int, error) { ... }
```

## 7. Тестирование

### Именование тестов

```
TestФункция_Сценарий_ОжидаемыйРезультат
```

```go
func TestUserCreator_DuplicateEmail_ReturnsError(t *testing.T) { ... }
func TestGetMe_RequiresAuth(t *testing.T) { ... }
func TestRightsDescribeFactory_ByRoles(t *testing.T) { ... }
```

### Структура тестов

```
tests/
  unit/           # Одна функция/метод, без I/O — всегда быстрые
  integration/    # Реальная БД (требуется запущенный Postgres)
  application/    # End-to-end HTTP (поднимается полный роутер)
```

### Правила

- Тесты используют `github.com/stretchr/testify` (`assert`, `require`).
- Моки — через интерфейсы: ручные или `gomock`. Никаких конкретных инфра-зависимостей в `unit/`.
- `internal/infrastructure/persistence/postgres/db/` — не тестируется напрямую (генерированный код).
- Table-driven тесты (`[]struct{ name, input, want }`) предпочтительны для множества кейсов.
- Пакет тестов: `package <pkg>_test` (чёрный ящик) — исключение только при необходимости доступа к неэкспортируемым символам.

### Покрытие

| Область | Минимум |
|---|---|
| Все публичные функции доменного слоя | 1 тест |
| Критические пути (auth, создание пользователя) | 90 % |
| Генерированный код (`db/`) | Не тестируется |

## 8. Формат коммитов

Используется [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>(<scope>): <краткое описание>

<тело: что и почему, не как — опционально>

<футер: breaking changes, ссылки на issues — опционально>
```

### Типы

| Тип | Когда |
|---|---|
| `feat` | Новая функциональность |
| `fix` | Исправление бага |
| `docs` | Только документация |
| `refactor` | Рефакторинг без изменения поведения |
| `test` | Добавление или правка тестов |
| `chore` | Сборка, CI, зависимости, тулинг |
| `perf` | Оптимизация производительности |

### Примеры

```
feat(auth): добавить эндпоинт обновления JWT-токена

Реализует автоматическое обновление access-токена по refresh-токену.
Refresh-токены хранятся в HTTP-only cookies.

Closes #42
```

```
fix(user): обработать nil-ответ репозитория при сохранении

Репозиторий может вернуть nil id при конфликте уникального индекса.
Добавлена проверка с возвратом ErrUserNotPersisted.
```

### Заголовок PR

Тот же формат, что и коммит: `<type>(<scope>): <описание>`


## 10. Коммуникация с агентом

### Структура задачи для агента

- Чёткий summary: что должно быть на выходе.
- Список файлов для создания или изменения.
- Acceptance criteria — формулировать как проверяемые условия.
- Negative constraints — явно указать, что задача **не** делает.
- Владельцы файлов — для параллельной работы без конфликтов.

### Ожидания от ревью

- Код агента проходит те же стандарты ревью, что и код человека.
- Ревью — на корректность и соответствие архитектуре, а не на стиль (стиль — задача линтера).
- Каждое замечание доводится до терминального состояния: исправлено, принято или отложено с обоснованием.

### Файлы контекста

| Файл | Назначение |
|---|---|
| `CLAUDE.md` | Контекст агента: структура, процессы, конвенции |
| `STYLE.md` | Соглашения о стиле и оформлении (этот файл) |
| `README.md` | Публичная документация: CLI-команды, эндпоинты |
| `FAQ.md` | Частые вопросы и решения |

Устаревший `CLAUDE.md` — источник деградации качества агента. Обновлять синхронно с изменениями в коде.

## 11. Жёсткие запреты

- **Никаких секретов и credentials в коде** — всё через переменные окружения (`.env` + `godotenv`).
- **Никаких `TODO` без ссылки на issue** — `// TODO(#42): рефакторить после релиза`.
- **Никаких заглушённых ошибок без комментария** — `_ = err` требует явного объяснения рядом.
- **Никакого `log.Fatal` / `os.Exit` вне `main()`** — нарушает инструментацию ошибок.
- **Никакого глобального состояния** — DI собирается только в `config/container.go`.
- **Никаких внешних импортов в `internal/domain/`** — домен изолирован от инфраструктуры.
- **Не редактировать вручную** `docs/` и `internal/infrastructure/persistence/postgres/db/` — генерируются через `make swag` и `make sqlc`.
- **Никаких wildcard-импортов** — импортировать только то, что используется.
- **Никакой бизнес-логики в `presentation/`** — только маппинг запроса → команда/запрос → ответ.
