## Находки

### 🔴 Критические / Высокие

#### 1. Тест `TestSearchAirports_CacheControlHeader` падает — CI-gate сломан *(новое)*
**Файл:** `tests/application/airports_search_http_test.go:164`  
**Описание:** `assert.Equal(t, "public, max-age=3600", rr.Header().Get("Cache-Control"))` — но хендлер этот заголовок **не устанавливает**. Тест падает при каждом запуске.  
**Сценарий отказа:** `go test ./tests/application/...` → `FAIL: expected "public, max-age=3600" actual ""` (воспроизведено локально). Исправление находки #2 из v1 (удаление заголовка) привело к регрессии: тест был добавлен уже после удаления заголовка и кодирует старое уязвимое поведение как ожидаемое. Либо тест надо удалить, либо заменить ожидание на `Cache-Control: private, max-age=3600` и вернуть заголовок в хендлер с безопасной директивой.

#### 2. SELECT в `CityRepository.Upsert` не проверяет тип ошибки — любой сбой БД маскируется *(новое)*
**Файл:** `internal/infrastructure/persistence/postgres/repository/city_repository.go:41`  
**Описание:** `if err == nil { return id, nil }` — если SELECT вернул не `pgx.ErrNoRows`, а реальную ошибку (обрыв соединения, таймаут), код молча продолжает и вызывает INSERT. Оригинальная ошибка теряется.  
**Сценарий отказа:** Кратковременный сбой коннекта во время SELECT → `err != nil` (не ErrNoRows) → код падает в INSERT → второй INSERT ещё раз возвращает ошибку, уже с текстом "insert city", а не исходный "context deadline exceeded". Диагностика по логам невозможна. При конкурентном повторе возможно `UNIQUE CONSTRAINT` на вставку одной и той же строки.

---

#### 3. TOCTOU-гонка в `CityRepository.Upsert` — параллельные синки дублируют строки *(новое)*
**Файл:** `internal/infrastructure/persistence/postgres/repository/city_repository.go:29`  
**Описание:** Логика SELECT → INSERT без `ON CONFLICT`. `CountryRepository` и `AirportRepository` используют настоящий upsert (`ON CONFLICT DO UPDATE`). При двух параллельных вызовах оба пройдут SELECT (0 строк) и оба попытаются INSERT.  
**Сценарий отказа:** Два параллельных `sync-airports` (или повтор после сбоя) → оба читают 0 строк для города → оба вставляют → второй получает `ERROR: duplicate key value violates unique constraint "cities_pkey"` → весь sync падает с частично записанными данными. Заменить SELECT+INSERT на `INSERT ... ON CONFLICT (lower(name), country_iso2, lower(state)) DO UPDATE SET timezone = EXCLUDED.timezone RETURNING id`.

#### 4. Аэропорты на уровне моря теряют высоту при каждом синке *(новое)*
**Файл:** `internal/application/command/sync_airports/handler.go:197`  
**Описание:** `if r.Elevation != 0 { e := r.Elevation; elevPtr = &e }` — высота 0 ft трактуется как «нет данных», поэтому `elevation_ft` записывается как NULL.  
**Сценарий отказа:** Аэропорты на уровне моря (NSTU/Паго-Паго, многие тихоокеанские острова, аэропорты в устьях рек) имеют elevation = 0. После каждого синка их `elevation_ft` перезаписывается NULL. Поисковый ответ возвращает `"elevation_ft": null` вместо `0`. Исправление: использовать отдельный булев флаг или разрешить 0 как валидное значение (убрать условие, всегда устанавливать указатель).

### 🟡 Средние *(перенесено из v1)*

#### 5. Порядок `lat`/`lon` инвертирован в `pool.Exec` — ловушка для будущих разработчиков
**Файл:** `internal/infrastructure/persistence/postgres/repository/airport_repository.go:83`  
**Описание:** Сигнатура `Upsert(... lat, lon float64 ...)`, но `pool.Exec` передаёт `(lon, lat)`. SQL компенсирует: `location[1] AS lon, location[2] AS lat`. Round-trip сейчас корректен, но контракт функции нарушен.  
**Сценарий отказа:** Будущий прямой вызов `Upsert(lat, lon)` «по документации» сохранит координаты транспонированными глобально — без ошибки компиляции. PostGIS-запросы или BI-инструменты, читающие `location[1]` как широту, получат долготу.

#### 6. Русское название города недетерминировано для многоаэропортных городов *(перенесено из v1 #5)*
**Файл:** `internal/application/command/sync_airports/handler.go:144`  
**Описание:** Русское название ищется по `r.ICAO` в `cityNamesRU`. Порядок итерации по Go-map случаен; если первый обработанный аэропорт города не имеет записи в Wikidata, город сохраняется на английском навсегда.  
**Сценарий отказа:** Москва: UUEE, UUWW, UUMO, UUBW. Если UUMO (без перевода) итерируется первым → город записывается как «Moscow». `seen`-guard срабатывает для остальных → «Москва» из Wikidata никогда не применяется. После синка поиск на русском не находит аэропорт.

#### 7. Wikidata-пагинация обрывается при неполной нефинальной странице *(перенесено из v1 #6)*
**Файл:** `internal/infrastructure/geo/wikidata/client.go:111`  
**Описание:** `if len(bindings) < pageSize { break }` — если Wikidata возвращает меньше 10 000 строк на промежуточной странице из-за серверного лимита, пагинация останавливается и оставшиеся данные молча теряются.  
**Сценарий отказа:** Wikidata возвращает 9 800 / 10 000 строк на странице N → break → все аэропорты после этого смещения выпадают из синка без ошибки. Синк завершается с кодом 0.

#### 8. Rate-limiter охватывает весь `/api/v1`, а не только `/airports` *(перенесено из v1 #7)*
**Файл:** `internal/presentation/http/router.go:85`  
**Описание:** `r.With(limiter.Handler).Route("/api/v1", ...)` применяет лимит к `POST /users`, `GET /users/me` и `GET /airports`. Комментарий и `Server.RateLimit` godoc говорят «per-IP cap for the airports endpoint».  
**Сценарий отказа:** Бот атакует `/api/v1/airports` → расходует токены лимитера → `/api/v1/users/me` получает 429 для легитимных пользователей на том же IP. Неаутентифицированные запросы (401) тоже расходуют токены.

#### 10. `mwgg`-клиент не имеет retry-логики в отличие от Wikidata-клиента *(новое)*
**Файл:** `internal/infrastructure/geo/mwgg/client.go:43`  
**Описание:** Один `http.Do` без повторов. `wikidata/client.go` реализует экспоненциальный backoff (4 попытки: 3 с, 6 с, 12 с). Асимметрия: транзиентный 503 от GitHub CDN роняет весь sync, хотя retry справился бы.  
**Сценарий отказа:** GitHub raw CDN возвращает 503 → `mwgg.Fetch` возвращает ошибку → весь `sync-airports` падает, несмотря на то что Wikidata-данные уже получены и готовы к использованию.
