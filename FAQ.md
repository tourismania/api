# FAQ


## 1. Зачем в `internal/infrastructure/persistence/postgres/db/users.sql.go` передается ctx?


`ctx` здесь выполняет сразу несколько важных функций:

**1. Отмена запроса (Cancellation)**
Если HTTP-обработчик завершился раньше (клиент закрыл соединение, timeout сработал), контекст отменяется — и `QueryRow(ctx, ...)` прервёт запрос к базе ещё до того, как PostgreSQL вернёт результат. Без `ctx` запрос бы продолжал выполняться вхолостую.

**2. Таймауты**
Если где-то выше по стеку сделали `context.WithTimeout(ctx, 5*time.Second)`, этот дедлайн автоматически распространяется на запрос к БД. Не нужно отдельно настраивать таймаут на уровне SQL.

**3. Трассировка и телеметрия**
Библиотеки вроде OpenTelemetry или pgx-instrumentation умеют вытаскивать из контекста span/trace ID и добавлять SQL-запрос как дочерний span. Это даёт видимость: сколько времени именно этот запрос занял внутри конкретного HTTP-запроса.

**4. Передача значений (request-scoped data)**
В контекст можно положить, например, user ID аутентифицированного пользователя или request ID — и middleware для логирования/аудита может их оттуда достать.

**Коротко:** `ctx` — это не просто формальность. Это механизм, который связывает жизненный цикл SQL-запроса с жизненным циклом HTTP-запроса, давая возможность вовремя отменить, ограничить по времени и проинструментировать обращение к БД.
